## 第一部分　平台是什么

### 1.1 一句话定位

**集团穿透式监管平台**是一套面向大型央企集团（新兴际华集团）的"**一平台三中心**"监管信息系统，目的是让集团对全级次子企业、全链条业务做到"**看得见、穿得透、调得动、管得住**"。

### 1.2 为什么需要这个平台

传统集团监管有三个痛点：

1. **看不全**：集团下属有板块、二级、三级甚至更多层级，财务、资金、投资、合同、安全等数据散落在不同系统里，集团层面很难一眼看全。
2. **穿不透**：一笔大额资金从集团账户到三级公司账户再到对手方账户，中间链路长，传统报表只能看到汇总数，看不到一笔笔凭证。
3. **调不动**：发现风险后，要打电话、发邮件、催整改，进度难追踪，闭环慢。

本平台通过**统一数据采集**把数据汇到一处、通过**规则引擎 + 关系图谱**自动识别风险、通过**工单状态机 + 指挥大屏**把处置流程固化下来，形成"**采集 → 监督 → 调度**"的闭环。

### 1.3 平台要解决的"四大穿透"

| 穿透维度 | 含义 | 例子 |
|---------|------|------|
| 全级次组织穿透 | 集团 → 板块 → 二级 → 三级 → 账户/凭证 | 从集团总账一路点到一个三级公司的某笔流水 |
| 全链条业务穿透 | 一笔业务从发起到入账的全过程 | 一笔投资决策 → 资金拨付 → 合同 → 凭证 |
| 全过程时间穿透 | 历史可回溯 | 看 3 年前某笔交易的全过程 |
| 全要素对象穿透 | 人 / 财 / 物 / 合同 / 项目都纳入 | 同一个对手方在不同业务里出现 |

### 1.4 覆盖的"十大监管领域"

平台围绕 10 个领域做风险识别与监管：财务资金、投资决策、合规风控、安全生产、采购、产权、人力、合同、关联交易、信息化（具体领域可在监管总览页"十大领域"卡片看到）。

## 第二部分　平台是怎么设计的

### 2.1 顶层架构：一平台三中心

```
┌─────────────────────────────────────────────────────────────┐
│  前端层  React 18 + TS + Vite + Zustand + Recharts + Tailwind │
│         (HashRouter, 17 个页面, 暗/亮双主题, 多端适配)         │
└──────────────────────────┬──────────────────────────────────┘
                           │ fetch /api/v1  (vite proxy → 7077)
┌──────────────────────────▼──────────────────────────────────┐
│  后端层  Fastify + TS  (端口 7077, 前缀 /api/v1)              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ 统一技术中台  JWT 鉴权 · RBAC · 行级权限 · 限流 · CORS    │ │
│  │              审计日志 · Prometheus 指标 · 健康检查        │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────────────┐  │
│  │ 数据采集中心  │ │ 智慧监督中心  │ │   调度指挥中心        │  │
│  │ 数据源 CRUD   │ │ 规则引擎      │ │ 工单状态机            │  │
│  │ 采集任务编排  │ │ 风险预警      │ │ (verify→rectify→      │  │
│  │ (全量/增量/CDC│ │ 关系图谱 BFS  │ │  review→archive)      │  │
│  │  三模式)      │ │ 穿透查询树    │ │ 自动派单 · 指挥大屏    │  │
│  │ 数据质量校验  │ │ 监管态势聚合  │ │ 超时检查              │  │
│  └──────────────┘ └──────────────┘ └──────────────────────┘  │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  AI 能力 / 数据脱敏  可插拔 LLM · 字段级脱敏管道         │ │
│  └─────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │  SQLite 数仓  ODS / DWD / DWS / ADS 分层                 │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
        │ 进程内 EventBus (RocketMQ 等价) 解耦三中心异步流转
```

**通俗解释**：
- **前端层**：你在浏览器里看到的页面（17 个），负责展示和交互。
- **后端层**：在服务器上跑的程序，负责算、存、鉴权。
- **统一技术中台**：后端的"门卫 + 管家"，统一管登录、权限、限流、审计、监控。
- **三中心**：平台的"三大业务模块"，下面单独讲。
- **AI 能力**：可选模块，提供自然语言查询、合同审查、报告生成，**所有数据进 AI 前必先脱敏**。
- **SQLite 数仓**：所有数据按 ODS（原始层）→ DWD（明细层）→ DWS（汇总层）→ ADS（应用层）四层分库存放，等价于 Apache Doris 的分层思想。

### 2.2 三中心各自干什么

#### ① 数据采集中心 —— "把数据汇过来"

- **数据源管理**：登记各个业务系统（浪潮 iGIX ERP、司库系统、Oracle 等）的连接方式。
- **采集任务编排**：支持**全量**（一次性拉全表）、**增量**（按时间戳拉新数据）、**CDC**（监听数据库 binlog 实时同步）三种模式。
- **数据质量校验**：采集后自动校验完整性、一致性。
- **采集趋势**：看每日采集量、延迟、异常数。

#### ② 智慧监督中心 —— "把风险找出来"

- **规则配置**：用规则 DSL（一种配置化的规则语言）写规则，比如"单笔资金流水 > 5000 万且对手方为关联企业 → 高风险"。
- **规则推理**：新数据进来后，规则引擎自动跑，命中即生成风险预警。
- **风险预警**：按领域、等级、状态筛选查看，每个预警有详情抽屉。
- **关系图谱**：账户 - 交易对手 - 组织 - 人员四级关联，BFS（广度优先）查找二度关联。
- **穿透查询**：集团 → 板块 → 二级 → 三级 → 账户/凭证/流水逐级下钻。
- **监管总览**：KPI、三中心健康度、十大领域、风险目录、保障、3 张图表的聚合页。

#### ③ 调度指挥中心 —— "把风险处置掉"

- **核查工单**：风险预警派单后变成工单，按 `verify（核查）→ rectify（整改）→ review（复核）→ archive（归档）` 四节点流转。
- **自动派单**：高级别预警通过 EventBus 自动派给对应领域的责任人。
- **指挥大屏**：16:9 大屏，KPI / 热力图 / 待办统计 / 资金流向 / 投资概览，30 秒自动刷新。
- **超时检查**：工单超时会自动催办。

### 2.3 三中心如何串成闭环

```
配置规则 → 规则推理命中 → 生成风险预警(pending)
            │
            └─EventBus─→ 高级别预警自动派单 → 工单(verify 节点)
                                                 │
                          ┌──────────────────────┘
                          ▼
              verify → rectify → review → archive
                          │
                          └─归档时回写风险预警为 resolved
```

**一句话**：数据采集中心喂料 → 智慧监督中心算出风险 → 调度指挥中心把风险处置掉并回写状态，三者通过 EventBus（事件总线）异步解耦。

### 2.4 技术中台提供什么保障

| 能力 | 等价生产组件 | 作用 |
|------|------------|------|
| JWT 鉴权 | Keycloak | 登录拿 token，后续请求带 token 验证身份 |
| RBAC + 行级权限 | Apache Ranger | 5 类角色 + 按组织层级过滤数据（二级单位只能看本单位及下级） |
| 限流 / CORS / 请求日志 | APISIX | 防止恶意刷接口、跨域、留痕 |
| 审计日志 | 自研 | 所有查询 / 处置 / 登录 / 脱敏 / AI 调用全留痕 |
| Prometheus 指标 | HertzBeat | `http_requests_total`、`risk_evaluations_total` 等指标暴露 |
| 健康检查 | HertzBeat | `/health` 接口无需鉴权，探活用 |

### 2.5 沙箱轻量实现 vs 生产级组件

平台当前用**同语义的轻量开源实现**承载方案的分布式组件，业务模型 / API 契约 / 数据流转完全一致，未来可平滑替换：

| 方案组件 | 中心 | 本期实现 |
|---------|------|---------|
| Airbyte / DataX / Debezium | 数据采集 | `collection/` + `node-cron` + 模拟连接器 |
| Apache Doris | 智慧监督 | SQLite + 分层视图 |
| NebulaGraph | 智慧监督 | 内存邻接表 + SQLite |
| Drools | 智慧监督 | `json-rules-engine` |
| LangChain | 智慧监督 | `ai/llm-adapter.ts` 可插拔适配器 |
| Apache Ranger（脱敏） | 技术中台 | `ai/sanitizer.ts` 字段级脱敏管道 |
| Flowable | 调度指挥 | `dispatch/workflow.ts` 工单状态机 |
| Apache Superset | 调度指挥 | 大屏聚合接口 + Recharts |
| RocketMQ | 调度指挥 | 进程内 EventBus |
| DolphinScheduler | 调度指挥 | `node-cron` |
| APISIX | 技术中台 | Fastify 插件 |
| Keycloak | 技术中台 | `@fastify/jwt` + RBAC |
| HertzBeat / Prometheus | 技术中台 | `/health` + `/metrics` |

> 这意味着：**今天能跑的业务流程，明天换成 Doris / NebulaGraph / Flowable 也跑得通**，只是底层换了引擎，前端、API、数据流转都不变。

### 3.2 本地启动（前后台联调）

**环境要求**：

| 依赖 | 版本 | 说明 |
|------|------|------|
| Node.js | ≥ 20 | 后端 `engines.node >= 20` |
| pnpm | ≥ 10 | 包管理（推荐） |
| Python / make / g++ | 系统自带 | 仅 `better-sqlite3` 原生编译需要 |

> Windows 用户建议在 WSL2 中运行，避免原生模块编译问题。

**一键启动步骤**：

```bash
# 1. 克隆代码
git clone https://github.com/mimi202605/Penetrating-Supervision-Platform.git
cd Penetrating-Supervision-Platform

# 2. 安装根工程依赖（前端 + concurrently）
pnpm install

# 3. 安装后端依赖（含 better-sqlite3 原生编译）
pnpm --dir server install

# 4. 一键并发启动前后台
pnpm dev:all
```

启动成功后：
- 后台 → <http://localhost:7077>（Fastify + tsx 热重载）
- 前台 → <http://localhost:5173>（Vite HMR）
- Vite 已自动把 `/api` 代理到 7077，前后台联调打通

## 第四部分　17 个页面逐个讲

### 4.1 监管总览 `/`（首页，最常打开）

**这是整个平台的"驾驶舱"**，1:1 还原设计稿，包含 12 个内容区块，从上到下依次是：

| 区块 | 看什么 | 用来干嘛 |

| 核查工单处置表（WorkOrderTable） | 7 列表格，含迷你步骤条 + 进度条 | 看工单进度 |

**交互**：

## 第五部分　典型业务闭环（最重要的部分）

这是平台的核心价值——**从发现风险到处置归档的完整流程**。请务必读懂这一节。

### 5.1 完整闭环图

```
   ┌─────────────────────────────────────────────────────────┐
   │                                                         │
   │  1. 数据采集中心把业务数据采进来                          │
   │      ↓                                                  │
   │  2. 智慧监督中心规则引擎跑规则                            │
   │      ↓                                                  │
   │  3. 命中 → 生成风险预警(pending)                          │
   │      ↓                                                  │
   │  4. EventBus 推送 → 高级别预警自动派单                    │
   │      ↓                                                  │
   │  5. 调度指挥中心生成工单(verify 节点)                     │
   │      ↓                                                  │
   │  6. 工单流转：verify → rectify → review → archive        │
   │      ↓                                                  │
   │  7. 归档时回写关联风险预警为 resolved                     │
   │      ↓                                                  │
   │  8. 监管总览 KPI / 态势图自动更新                         │
   │                                                         │
   └─────────────────────────────────────────────────────────┘
```

### 5.2 各角色在闭环里干什么

| 角色 | 在哪一步做什么 |
|------|------------|
| 集团监管员（admin） | 配规则、看总览、派单、复盘 |
| 二级单位监管员（group2） | 看本单位风险、处置本单位工单 |
| 业务核查人员（inspector） | 在工单 verify/rectify/review 节点填处置结果、推进工单 |
| 调度值班员（duty） | 盯指挥大屏、催超时工单 |
| 集团领导（leader） | 看总览和大屏，只读 |

### 5.3 用 API 走一遍闭环（命令行操作）

后端提供完整 API，可以用 curl 走一遍闭环（统一前缀 `/api/v1`）：

```bash
# 步骤 1：登录拿 token
curl -X POST http://localhost:7077/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123"}'
# 返回 { "token": "xxx", "user": {...} }

TOKEN="xxx"  # 替换为上一步拿到的 token

# 步骤 2：规则推理命中（生成风险预警 + 自动派单）
curl -X POST http://localhost:7077/api/v1/monitoring/rules/rule-001/evaluate \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"amount": 50000000, "counterparty": "Everwin Holdings"}'
# 这一步会：
#  ① 规则引擎命中 → 生成 risk_warning (pending)
#  ② 如果是高级别 → EventBus 自动派单 → 生成 work_order (verify 节点)

# 步骤 3：工单节点流转（核查 → 整改 → 复核 → 归档）
curl -X POST http://localhost:7077/api/v1/dispatch/work-orders/wo-001/advance \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"result": "已核实，需整改", "operator": "inspector"}'
# 重复 advance 直到 archive，归档时自动回写风险预警为 resolved
```

### 5.4 一键验证闭环

后端提供了一键脚本，自动跑完整闭环：

脚本会自动执行：规则评估 → 自动派单 → 工单流转 → 归档 → 风险回写 resolved，并打印每一步的结果，用来验证三中心联动是否正常。

## 第六部分　AI 能力与数据脱敏

### 6.1 AI 能力总览

平台提供 3 个 AI 能力（**全部可选**，未配置 `AI_API_BASE` 时返回结构化占位响应）：

| 能力 | 端点 | 用途 |
|------|------|------|
| 自然语言穿透查询 | `POST /ai/query` | 用自然语言问问题，AI 转成查询并返回结果 |
| 合同违规条款审查 | `POST /ai/contract-review` | 上传合同，AI 自动识别违规条款 |
| 风险处置报告生成 | `POST /ai/risk-report` | 基于风险预警 + 工单数据自动生成报告 |

### 6.2 如何接入 AI

```bash
# 对接集团微调 Llama3 或国产大模型，只要 OpenAI 兼容协议即可
export AI_API_BASE=https://your-llm-gateway/v1
export AI_API_KEY=sk-xxxx
export AI_MODEL=llama3-70b-instruct

# 重启后端，/api/v1/ai/** 将由占位响应切换为真实 LLM 调用
```

### 6.3 数据脱敏（最重要，必读）

**核心原则：原始敏感数据零外送。**

所有业务数据传入 LLM 之前，**必须**经过 `sanitizeForAI(payload, policy)` 字段级脱敏管道，支持 4 种算法：

| 算法 | 说明 | 例子 |
|------|------|------|
| mask | 掩码（保留首尾，中间打*） | `张三丰` → `张*丰` |
| hash | 哈希（不可逆） | `622848` → `a3f9...` |
| replace | 替换为占位符 | `北京市朝阳区` → `[地址]` |
| range | 区间化 | `50000000` → `1000万-1亿` |

**保障措施**：
- 脱敏事件落审计表（可追溯谁、什么时候、对什么数据做了脱敏）
- AI 调用全链路写 `ai_call_logs`：入参脱敏后摘要 / 出参摘要 / 耗时 / token / 调用者
- 脱敏策略可 CRUD（`/ai/sanitizer/policies`）

**为什么这么做**：央企数据敏感，一旦原始数据传给外部 LLM 就有泄露风险。脱敏后即使 LLM 服务商拿到数据也无法还原。

### 6.4 用 AI 做一次自然语言查询

```bash
curl -X POST http://localhost:7077/api/v1/ai/query \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"query": "查看新兴铸管 2026 年三季度大额资金流向"}'
```

**后台发生的事**：
1. 接到请求，识别敏感字段（公司名、金额）
2. 调 `sanitizeForAI` 脱敏
3. 脱敏后摘要写 `ai_call_logs`
4. 调 LLM（OpenAI 兼容协议）
5. 拿到结果，出参摘要写 `ai_call_logs`
6. 返回给前端

### 7.2 行级权限是怎么做的

二级监管员（group2）登录后，**所有查询自动加组织层级过滤**：
- 只能看到本单位及下级的数据
- 集团层面的数据看不到
- 这是技术中台的 Ranger 等价实现，在 SQL 层自动加 `WHERE org_id IN (...)`

**好处**：业务代码不用关心权限，写一次到处跑；权限调整只改组织树即可。

- 后续请求自动带 `Authorization: Bearer <token>` 头
- token 过期需重新登录
- 登出会落审计日志

## 第八部分　常见问题与排错

### Q1：登不上，提示 401

- 检查用户名密码是否正确（种子用户密码统一为 `admin123`）
- 检查后端是否启动（访问 <http://localhost:7077/health> 应返回 ok）
- 检查 token 是否过期，重新登录

### Q2：页面打开了但看不到数据

- **本地模式**：确认后端已启动（`pnpm dev:all` 而非 `pnpm dev`）
- 检查浏览器控制台是否有 CORS 错误
- 检查 `VITE_USE_MOCK` 环境变量：`false` 走真实后端，`true` 走 Mock
- 检查 vite 代理是否生效（`/api` → 7077）

### Q3：演示站点打不开

- 演示站点为 GitHub Pages 纯静态托管，**不需要后端**
- 直接访问 <https://mimi202605.github.io/Penetrating-Supervision-Platform/>
- 演示站点是 Mock 模式，数据是预设的，不能改

### Q4：better-sqlite3 编译失败

- Windows 用户请在 WSL2 中运行
- 确保 Node.js ≥ 20
- 确保系统有 Python / make / g++（Linux 一般预装，macOS 需 xcode-select --install）

### Q5：AI 查询返回"占位响应"

- 说明未配置 `AI_API_BASE`
- 按 6.2 节配置 `AI_API_BASE` / `AI_API_KEY` / `AI_MODEL`
- 重启后端

### Q6：工单推进不了

- 检查当前用户是否有处置权限（inspector 角色）
- 检查工单是否已 archived（归档后不能再推进）
- 检查是否填了 `result` 字段（advance 接口必填）

### Q7：风险预警一直是 pending

- 检查是否高级别（high）—— 只有 high 才自动派单
- 中低级别需手动派单
- 检查 EventBus 是否正常（看后端日志）

### Q8：如何验证三中心闭环正常

```bash
bash server/scripts/closed-loop.sh
```

一键验证：规则评估 → 自动派单 → 工单流转 → 归档 → 风险回写 resolved。

